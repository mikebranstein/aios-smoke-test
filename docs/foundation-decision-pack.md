## 1. Project Overview
Skyline Rhythm is a PC-first movement-rhythm game focused on jump-rope timing, trick expression, and flow in both single-player and co-op play. The MVP architecture prioritizes crisp input feel, beat-readable gameplay, and modular systems for scoring, progression, reactive audio/visual feedback, and mode-specific rule sets. The technical baseline is chosen to let feature teams build immediately while isolating online and live-ops uncertainty behind clear interfaces.

## 2. Core Technical Decisions

### 2.1 Architecture Topology
Locked: **Modular monolith game client** with explicit domain modules (Input/Timing, Rope Simulation, Trick Grammar, Scoring/Combo, Progression, Mode Rules, Audio Reactivity, UI/HUD, Online Adapters). Rationale: one deployable keeps iteration speed high for MVP while preserving clean seams for later service extraction if online scale requires it.

### 2.2 Platform / Runtime
Locked: **PC runtime target (Windows as primary ship target, Steam Deck/Linux compatibility as secondary)**. Rationale: this aligns with the foundation scope of PC delivery while keeping performance and input latency tuning focused on one primary runtime first.

### 2.3 Language & Framework Stack
Locked: **Unity 6 LTS + C#** for gameplay, UI, and tooling; package boundaries map to the architecture modules above. Rationale: mature cross-platform tooling, strong iteration speed for movement-heavy gameplay, and broad ecosystem support for rhythm timing, animation, and content pipelines.

### 2.4 State & Persistence
Locked: **Local-first persistence** using versioned save files (JSON/binary) for player profile, progression, settings, and replay metadata, wrapped by a SaveRepository interface. Rationale: supports offline-first MVP development and deterministic local testing while allowing later cloud sync or backend persistence via adapter replacement.

### 2.5 Identity / Account / Tenancy
Locked: **Single-player local profile model** for core game progression; online modes bind to a platform account ID when connected `(inferred)`. Rationale: avoids account-system overbuild during MVP while providing a clear identity key for co-op sessions and asynchronous rankings.

### 2.6 API / Integration Contract
Locked baseline: **Versioned HTTPS JSON APIs** for online features (session matchmaking metadata, leaderboard submission/query, seasonal metadata) behind an OnlineGateway interface in the client.

## 3. Guardrails for Autonomous Agents

### 3.1 Agent Autonomy Boundary
Agents may autonomously implement changes **inside existing module boundaries** (bug fixes, feature slices, tests, refactors that do not alter external contracts), add non-breaking fields, and improve performance without changing gameplay rules. Agents must request human/ADR review before changing engine/runtime, introducing new network protocols, changing persistence format compatibility guarantees, altering scoring fairness rules, adding new external services, or changing security/privacy data handling.

### 3.2 Forbidden Patterns
Do not introduce: cross-module circular dependencies, frame-rate-coupled gameplay timing logic, client-authoritative leaderboard acceptance, hardcoded secrets/keys, silent exception swallowing on save/load/network paths, or direct online SDK calls outside the OnlineGateway abstraction.

### 3.3 Required Conventions
Use deterministic timing abstractions (fixed-step or explicit clock injection) for gameplay-critical systems; keep mode rules data-driven where feasible; include unit tests for scoring/timing logic and integration tests for save/load compatibility; document any contract change in module README/ADR notes in the same PR.

### 3.4 Dependency Policy
New dependencies must come from approved registries (Unity Package Manager official registry or vetted Git sources), use permissive licenses (MIT/BSD/Apache-2.0), and avoid copyleft obligations that would constrain distribution. Any networking, analytics, or user-content dependency requires human approval before merge.

### 3.5 Migration Policy
Persistence and API changes must be backward compatible for at least one minor release cycle: add version fields, provide explicit migration code paths, and keep old readers/endpoints active during transition. Breaking schema/API changes require ADR + rollout plan + rollback path before implementation.
